# AI-генератор еженедельного отчета

Вы руководите продуктовой/сервисной командой внутри банка.

Ваша команда получает обращения от внутренних клиентов: заказчика, операционных подразделений, смежных комманд, комплаенса, штаба и т.д.

Нужно при помощи vibecoding собрать инструмент, который автоматически анализирует обращения, задачи команды и метрики продукта, а затем генерирует еженедельный отчет для вас как для руководителя.

На вход подаются обращения внутренних клиентов за месяц, задачи команды из спринта и бэклога, а также данные из SQL.

Решение должно автоматически очищать данные, подсчитывать метрики, выявлять проблемные зоны, генерировать executive summary и предложить оптимальное перераспределение задач и обращений между участниками команды.

# DoD:

Инструмент должен:

1. Загрузить данные из CSV, XLSX и SQL.
2. Очистить и нормализовать данные.
3. Посчитать ключевые метрики за последнюю неделю.
4. Сравнить последнюю неделю с предыдущей.
5. Найти проблемные зоны:
    - рост обращений;
    - нарушение SLA;
    - перегруженные исполнители;
    - частые повторные открытия;
    - просроченные задачи;
    - блокировки.
6. Сгенерировать еженедельный отчет.

---

# Обязательные метрики отчета

## По обращениям

- количество обращений за неделю;
- динамика к прошлой неделе;
- топ-5 подразделений-инициаторов;
- топ-5 типов обращений;
- доля нарушений SLA;
- среднее время решения;
- количество повторно открытых обращений;
- средняя оценка внутренних клиентов;
- проблемные продуктовые области.

## По задачам команды

- количество задач в текущем спринте;
- количество просроченных задач;
- количество заблокированных задач;
- загрузка участников команды;
- задачи без исполнителя;
- задачи, связанные с обращениями;
- распределение задач по типам.

## По участникам команды

- текущая загрузка;
- количество назначенных обращений;
- количество задач;
- суммарная оценка трудозатрат;
- превышение capacity;
- зоны специализации.

---

## AI summary

Система должна сгенерировать:

1. **Executive summary** на 5–8 предложений.
2. **Основные риски недели**.
3. **Причины роста обращений**, если рост есть.
4. **Ключевые проблемные темы из текстов обращений**.
5. **Рекомендации руководителю команды**.

---

# Оптимизационные предложения по распределению задач

## Логика рекомендаций

Учитывать:

- текущую загрузку участника;
- capacity в часах;
- дедлайны;
- приоритеты;
- SLA;
- продуктовую область;
- skills участника;
- тип задачи;
- наличие блокеров;
- бизнес-вес подразделения;
- критичность product area.

## Пример результата

```
Рекомендация 1:
Передать 3 обращения типа "ошибка в данных" по области "Отчетность" от Анны к Дмитрию.

Причина:
Анна загружена на 128% capacity, у Дмитрия загрузка 64%, а в skills Дмитрия указаны "reporting, data_quality".

Ожидаемый эффект:
Снижение риска нарушения SLA по high-priority обращениям.
```

```
Рекомендация 2:
Перенести задачу TASK-142 из текущего спринта в next sprint.

Причина:
Задача имеет priority=medium, не связана с критичными обращениями, при этом блокирует capacity senior-разработчика.

Ожидаемый эффект:
Освобождение 12 часов на закрытие urgent support-задач.
```

# Формат итогового отчета

Минимум один из вариантов:

- HTML-отчет;
- PDF;
- DOCX;
- PPTX;
- Streamlit dashboard + export.

Рекомендуемый формат для контеста:

```
weekly_report.html
```

или

```
weekly_report.pptx
```

---

# Структура отчета

1. **Executive summary**
2. **Ключевые цифры недели**
3. **Динамика обращений**
4. **SLA и качество сервиса**
5. **Нагрузка команды**
6. **Проблемные зоны**
7. **AI-анализ текстов обращений**
8. **Рекомендации по перераспределению задач**
9. **Список критичных действий на следующую неделю**

# Формат сдачи репозитория

```
solution/
	src/ # Кодовая база решения
  README.md
  output/
    weekly_report.html или weekly_report.pptx
```

README должен содержать:

```
1. Как запустить
2. Какие файлы нужны на вход
3. Где лежит результат
4. Какие AI-функции используются
5. Какие ограничения есть у решения
```
